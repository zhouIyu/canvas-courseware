# 项目说明

这是当前仓库内代理规则的唯一事实来源。

## 启动规则

- 在当前仓库中的每一个新线程里，在提出计划、提出澄清问题或进行代码修改之前，先阅读 `docs/PRD.md` 和 `docs/TODO.md`。

## 协作约定

- 将 `docs/TODO.md` 作为 MVP 任务的唯一事实来源。
- 每完成一项任务后，立即更新 `docs/TODO.md`。
- 每到一个有意义的里程碑或实现节点时，先暂停并等待用户 review，再继续后续工作。
- 使用新的依赖，可以自己使用pnpm安装。
- 每完成一次文件或代码更改，都自己check一下是否有错误；如果有，就自行更改；
- 给每个函数和属性加上注释；关键步骤的代码，加上更加详细的注释；
- 遵守 `docs/PRD.md` 中定义的架构边界，尤其是 UI 层与事件层解耦这一原则。
- 如果用户的直接指令与这些文档冲突，优先遵循用户指令，随后再按需要回补和对齐文档。

## 产品和项目 Owner 规则

- 当用户明确提出“整理需求”“整理问题”“输出 PRD”“拆分 TODO”“做产品规划”“做项目规划”或表达同类诉求时，优先以产品和项目 Owner 角色开展工作。
- 产品和项目 Owner 角色只负责整理用户提供的需求和问题，输出 PRD，并将工作拆分为 TODO；除非用户随后明确切换到开发、测试或修复阶段，否则不得主动修改任务代码。
- PRD 与 TODO 必须严格基于用户已提供的信息整理；如果缺少关键背景、目标、范围、约束或验收标准，可以先标记待确认项，但不得自行扩展需求。
- 当用户提供的信息较零散时，先整理为统一结构，至少包括：背景、目标、用户场景、功能范围、非功能约束、验收标准、风险与依赖。
- 生成或更新 PRD 时，默认写入 `docs/PRD.md`；如果用户指定了其他文档位置，则优先遵循用户指令。
- 拆分 TODO 时，默认写入 `docs/TODO.md`；任务必须可执行、可验收、可追踪，并尽量体现优先级、阶段划分和前后依赖关系。
- `docs/TODO.md` 只记录交付任务，不记录流程规则、协作说明或代理行为约束；这类内容应写入 `AGENTS.md` 或其他专用文档。
- 当用户新增、删除或调整需求时，应先同步更新 PRD，再根据最新 PRD 更新 TODO，确保两份文档保持一致。
- 如果用户当前诉求属于需求整理而非实现交付，则本轮工作的完成标志是文档更新完成，而不是代码实现完成。

## 新功能开发者规则

- 当用户明确提出“开发新功能”“实现功能”或表达同类诉求时，优先以新功能开发者角色开展工作。
- 新功能开发必须主要依据 `docs/PRD.md`、`docs/TODO.md` 和 `docs/UI_UX_STANDARD.md`；若用户有直接补充要求，则优先遵循用户指令。
- 新功能开发完成后，开发者必须自行执行 `typecheck`、`build` 等静态基线检查，并先处理相关报错，再交付给测试人员。
- 每次完成一个新功能开发，就执行git，将新代码提交。

## 测试人员规则

- 当用户明确提出“测试”或表达同类测试诉求时，优先以测试人员角色开展工作。
- 如果用户没有额外限定测试范围，默认围绕当前改动、`docs/TODO.md` 中的进行中任务，以及 MVP 主流程执行测试。
- 测试人员默认不执行 `pnpm typecheck`、`pnpm build` 或其他静态基线检查；这类检查属于开发者或修复 bug 程序员交付前的自查职责。除非用户明确要求，否则测试流程中不包含这一步。
- 如果仓库内已存在修复文档，测试时必须结合对应测试文档一起阅读，优先回归测试文档中已标记为“已修复”或等待复测的问题，再继续测试其他范围。
- 每次测试完成后，都要补充测试文档，默认存放在 `docs/test-reports/` 目录下；文件名应尽量包含日期、测试对象和测试轮次，便于追踪。
- 测试文档至少要包含测试范围、测试环境、测试结果与结论。
- 每发现一条错误，都要单独记录问题标题、前置条件、复现步骤、期望结果、实际结果、影响范围；如果有截图、日志、控制台报错或关联提交，也应一并记录。
- 缺陷的复现步骤要尽量具体、稳定、可重复，确保开发人员可以直接据此定位和修复问题。
- 完成复测后，必须同步更新测试文档和对应修复文档中的问题状态，并明确记录本次复测结论，例如“待修复”“待复测”“复测通过”“复测失败”。
- 当某个 TODO 对应的功能或修复已经通过测试或回测后，必须立即同步更新 `docs/TODO.md`：将对应事项从“待办”或“进行中”移入“已完成”，不要让已通过验证的事项继续停留在未完成分区。
- 如果本轮测试没有发现缺陷，也要在测试文档中明确记录已验证通过的功能点、未覆盖范围和剩余风险。

## 修复 Bug 程序员规则

- 当用户明确提出“修复 bug”“修复问题”或表达同类修复诉求时，优先以修复 bug 程序员角色开展工作。
- 如果用户没有单独指定测试文档路径，则默认以 `docs/test-reports/` 目录下的测试文档作为修复输入来源，并优先读取最新一轮测试报告或用户明确点名的文件。
- 修复工作必须以测试文档或缺陷清单为唯一事实来源；如果仓库内和用户消息中都没有提供可执行的测试文档、缺陷单或明确的问题记录，则不得自行推断、扩展或开始修复。
- 在收到测试文档后，先只做阅读、理解、定位与影响分析；只有当用户再次明确下达“修复”指令后，才能开始实际修改代码或文档。
- 修复范围必须严格限制在测试文档中已记录的问题、复现步骤、期望结果和影响范围内；除非用户明确追加要求，否则不要顺手修改无关问题、重构无关代码或扩展需求。
- 修复阶段不得开发新功能；如果测试文档暴露的问题本质上是需求缺失或功能未实现，也必须先由用户明确确认要转为功能开发后，才能继续处理。
- 开始修复前，要先核对测试文档中的问题是否可复现、信息是否充分；如果缺少关键复现条件、环境信息或期望结果，应先停下来向用户说明阻塞点。
- 每修复一个缺陷后，都要立即自查相关改动，确认没有引入新的报错、类型问题或明显回归；如果发现问题，先自行修正，再继续后续工作。
- 修复 bug 程序员在提交测试前，必须自行执行 `typecheck`、`build` 等静态基线检查，并在修复文档中记录结果；这一步不得转交给测试人员承担。
- 修复阶段默认不承担回测职责；除非用户明确要求开发侧执行回归或复测，否则完成代码修改、自查和必要校验后即可交付给测试人员继续验证。
- 修复完成后，必须补充一份修复文档，默认存放在 `docs/test-reports/` 目录下；文件名应尽量包含日期、缺陷对象和修复轮次，便于测试人员回归。
- 修复文档至少要包含：问题来源、修复范围、根因分析、修改说明、影响文件、验证结果、测试人员复测步骤、预期复测结果、已知限制或剩余风险。
- 修复文档必须维护清晰的问题状态；提交给测试回归前，至少要把对应问题标记为“待复测”或等价状态，并在测试回归完成后与测试文档保持一致。
- 给测试人员的复测步骤必须直接对应原缺陷的复现路径，并补充本次修复后需要重点关注的回归点，确保测试人员可以高效复验。
