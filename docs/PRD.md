# Canvas Courseware SDK MVP PRD

## 1. 产品概述

- 产品名称：Canvas Courseware SDK MVP
- 产品定位：一个面向课件编辑与播放场景的 TypeScript SDK，并提供基于 Vue 3 的参考实现。
- 产品目标：仿照 PPT 的核心编辑体验，支持多页 slide、单页独立 canvas、基础元素编辑、分步播放和动画触发。

## 2. 项目目标

### 2.1 业务目标

- 产出一套可嵌入业务系统的课件 SDK。
- 用一份通用 JSON 描述课件内容，不与 Vue 或 Fabric 绑定。
- 用一个 Vue 3 monorepo 工程交付 MVP，便于后续迭代为正式产品。
- 在 app 层提供一套可直接使用的课件创作工作台，作为 SDK 的正式产品化示例。

### 2.2 MVP 目标

- 支持 `Text`、`Image`、`Rect` 三类元素。
- 支持多页 slide，每页对应一个独立 canvas。
- 支持编辑模式与预览模式。
- 预览模式支持逐步出现、基础动画、点击触发。
- 编辑模式支持选中、拖拽、缩放、层级列表。
- 提供项目列表页、正式编辑器页和本地持久化保存。

## 3. 核心设计原则

### 3.1 UI 层与事件层解耦

- UI 层只负责展示状态与派发意图，不直接持有 Fabric 事件逻辑。
- 事件层独立维护命令、交互事件、状态变更和订阅机制。
- 渲染适配层负责把底层引擎事件转换成标准化事件，再交给事件层处理。
- 任意 UI 实现都应只依赖通用状态和事件协议，而不是依赖 Fabric 对象。

### 3.2 文档模型与引擎解耦

- 课件 JSON 不出现 Fabric、Konva、Pixi 专有字段。
- 节点、动画、步骤、图层顺序都以通用结构表达。
- 适配层只负责“文档模型 <-> 渲染对象”的映射。

### 3.3 编辑态与播放态共享一份模型

- 编辑模式和预览模式共用同一份文档模型。
- 播放态不持有另一套独立数据结构，只消费 slide 和 timeline。

## 4. 目标用户

- 需要在业务系统里嵌入课件编辑器的前端团队。
- 需要定制课件播放逻辑的互动课件团队。
- 需要在后续替换底层渲染引擎的 SDK 团队。

### 5.1 In Scope

## 6. 关键功能需求

### 6.1 文档结构

- 一个 `Document` 包含多个 `Slide`
- 一个 `Slide` 包含：
  - 画布尺寸
  - 背景配置
  - 元素列表
  - timeline 步骤列表
- 每个 slide 在编辑态和播放态都可独立渲染

### 6.2 元素能力

- `Text`
  - 内容
  - 字体大小
  - 颜色
  - 宽高与位置
- `Image`
  - URL
  - 宽高与位置
  - 基础适配方式
- `Rect`
  - 填充色
  - 描边
  - 宽高与位置

### 6.3 编辑模式

- 支持从项目列表进入课件工作台
- 工作台支持三栏布局：左侧导航/结构区、中央编辑区、右侧属性/时间轴区
- 支持 slide 切换
- 支持新增 Text/Image/Rect
- 编辑态默认展示当前 slide 的全部元素，不因 timeline 步骤而隐藏后续元素
- 支持选中元素
- 支持拖拽、缩放
- 支持属性面板修改基础属性
- 支持层级列表展示
- 支持上移、下移、置顶、置底
- 对属于后续步骤的元素，编辑态需提供明确的步骤归属提示，便于排版和时间轴配置

### 6.4 预览模式

- 支持按 slide 预览
- 支持在同一工作台内从编辑模式切换到预览模式
- 预览模式沿用工作台三栏结构，中央预览区自适应剩余空间
- 支持点击进入下一步
- 支持自动触发步骤
- 支持对象点击触发步骤
- 支持元素逐步出现
- 预览态按 timeline 步骤和 trigger 严格控制元素显隐与动画播放
- MVP 动画类型：
  - `appear`
  - `fade`
  - `slide-up`

## 7. 非功能要求

- 文档 schema 稳定、可扩展、可版本化
- SDK 代码按职责拆包，便于替换 UI 或渲染引擎
- Fabric 只存在于适配层和参考实现，不进入 schema 和事件协议
- 代码以 TypeScript 为主，尽量避免运行时魔法
- 工作台在常见桌面分辨率下应具备稳定的自适应布局能力，默认使用场景中不出现非预期的页面级滚动条或内容截断
- 工作台采用三栏布局时，左右侧栏宽度固定，中央编辑区/预览区自适应剩余宽度，并保持核心画布优先
- 工作台应优先保障主编辑区与预览区的垂直可用空间，顶部信息区和辅助面板不得过度挤占核心画布工作区高度
- 工作台信息架构应以画布编辑与预览为中心，避免仪表盘式的大面积统计卡片和过多平级功能分区，非核心信息应收敛、合并或折叠
- 工作台默认界面应控制信息密度，优先展示当前任务必需的信息与操作；统计、摘要、次要状态和低频操作应降级展示，避免页面显得拥挤和杂乱
- 工作台滚动交互应符合自然阅读和编辑习惯，纵向内容优先采用页面级滚动；主编辑区、预览区及其外层主容器不应依赖内嵌滚动完成主要浏览和操作
- 左右侧栏应支持一键收起与展开，收起后中央区域即时扩展，提升编辑区与预览区可用空间

## 8. 架构方案

### 8.1 分层设计

- `packages/core`
  - 通用 schema
  - store/state
  - command/event contracts
  - timeline helpers
- `packages/fabric`
  - Fabric 适配器
  - 渲染对象映射
  - Fabric 事件标准化
- `packages/vue`
  - Vue 3 参考 UI
  - 编辑器壳层
  - 预览器壳层
- `apps/playground`
  - MVP 正式应用壳层
  - 项目列表页
  - 编辑器工作台页
  - 本地项目持久化

### 8.2 数据流

1. UI 层派发用户意图，例如“选中节点”“更新文本”“调整层级”
2. 事件层把意图转换为标准命令
3. Store 更新文档状态
4. 渲染适配层订阅状态变化并同步到底层引擎
5. 底层引擎产生交互事件后，由适配层归一化回传事件层
6. UI 层订阅最新状态并刷新展示

### 8.3 解耦边界

- Vue 组件不能直接操作 Fabric 对象
- Fabric 事件不能直接修改 Vue 本地状态
- 任何元素属性变更都必须先进入标准事件/命令层
- 所有播放步骤都由通用 timeline 描述，而不是由 UI 写死

## 9. 通用 JSON 设计要求

### 9.1 必要结构

- `document`
- `slides[]`
- `nodes[]`
- `timeline[]`
- `steps[]`
- `animations[]`

### 9.2 设计要求

- 节点顺序可表达图层顺序
- timeline 以 step 为中心组织，step 归属于当前 slide，并支持 page-click / auto / node-click 等触发方式
- 当 step 使用对象点击触发时，需显式记录触发对象，而不仅是页面级点击
- animation 与 node 解耦，通过 `targetId` 关联
- schema 自带 `version`

## 10. 验收标准

- 能创建多页课件并切换 slide
- 能从项目列表进入某个课件项目
- 能新增 Text/Image/Rect
- 编辑态能看到当前 slide 的全部元素，并识别元素所属步骤
- 能对元素进行选中、拖拽、缩放
- 能看到并调整层级列表
- 能在编辑 / 预览模式之间切换
- 工作台采用三栏布局时，左右侧栏固定，中央区域可随视口与侧栏状态自适应伸缩
- 左右侧栏支持一键收起/展开，收起后中央编辑区或预览区可明显扩展
- 工作台在常见桌面视口下默认不出现非预期横向滚动条
- 工作台在内容区、侧栏和顶部工具区同时展示时，布局仍能完整容纳主要操作区域
- 工作台纵向浏览默认采用页面级滚动，而不是把主要内容限制在内嵌滚动容器中
- 如需局部滚动，仅应出现在明确设计的辅助区域中，且不影响主编辑区与预览区的连续浏览和操作
- 主编辑区与预览区在常见桌面视口下应保持足够的首屏可视高度，避免核心画布区域被顶部摘要信息过度压缩
- 在编辑态与预览态切换后，核心画布工作区仍应保持“以画布为主”的布局占比
- 非核心统计、状态和辅助信息区数量应被控制，默认界面不以多个大卡片分割主工作流
- 同一层级的功能分区应尽量收敛，避免因区域过多导致用户视线分散和操作路径拉长
- 默认首屏只展示当前编辑/预览所需的核心信息，次要统计与辅助说明不应占据大面积视觉空间
- 默认页面的信息块数量与样式层级应被压缩，避免同一屏出现过多抢占注意力的模块
- 时间轴新增步骤时，支持选择页面点击、自动触发或对象点击，而不是只支持页面级触发
- 当步骤配置为对象点击触发时，用户能明确选择触发对象并完成预览验证
- 能把项目保存到本地并在刷新后恢复
- 能导出一份通用 JSON
- 能基于该 JSON 进行预览播放
- 能通过点击或自动步骤触发元素按顺序出现

## 12. 下一版本规划（VNext）

### 12.1 版本定位

- 下一版本聚焦“创作效率与编排能力”，不再延续当前版本已进入修复中的工作台布局、滚动与信息精简问题。
- 版本目标是降低重复操作成本，补齐高频编辑能力，并让 slide 编排、图层管理、时间轴配置形成更完整闭环。
- 优先解决“做得慢、排版费劲、页面编排不顺、时间轴难维护、内容导入不方便”等核心体验问题。
- 下一版本的页面壳层与通用交互控件统一基于 Arco Design Vue 重构，建立一致的工作台 UI 语言与组件规范。

### 12.2 核心功能范围

- Arco Design 页面重构
  - `apps/playground` 与 `packages/vue` 的页面壳层、导航区、面板区、表单区、弹窗区、抽屉区、标签与按钮等通用 UI，统一采用 Arco Design Vue 组件体系重构。
  - 优先复用 Arco Design Vue 的布局、导航、表单、数据展示与反馈类组件，减少自定义基础界面组件维护成本。
  - 页面结构、视觉语言、交互反馈与状态表达，统一参考 Arco Design 的设计规范，避免工作台中同时存在多套风格。
  - 画布渲染区、底层事件交互、文档 schema 与 Fabric 适配层不依赖 Arco Design，仍保持现有架构边界与解耦原则。
- 撤销 / 重做
  - 覆盖新增、删除、移动、缩放、属性修改、层级调整、slide 管理、timeline 步骤编辑等主要操作。
  - 支持连续回退与恢复，保证高频编辑场景可逆。
- 快捷键体系
  - 支持 `Delete / Backspace` 删除。
  - 支持 `Cmd/Ctrl + Z` 撤销、`Cmd/Ctrl + Shift + Z` 或 `Cmd/Ctrl + Y` 重做。
  - 支持 `Cmd/Ctrl + C`、`Cmd/Ctrl + V`、`Cmd/Ctrl + D`。
  - 支持方向键微调位置，`Shift + 方向键` 做大步移动。
- 复制 / 粘贴 / 重复
  - 支持同页复制、粘贴、重复节点。
  - 支持跨 slide 粘贴节点。
  - 复制后的节点默认保留基础属性、样式与资源引用，并做适度位置偏移，避免完全重叠。
- 多选与批量排版
  - 支持框选与多选。
  - 支持批量移动。
  - 支持左对齐、水平居中、右对齐、顶对齐、垂直居中、底对齐。
  - 支持横向分布与纵向分布。
- Slide 管理增强
  - 支持复制 slide。
  - 支持 slide 拖拽排序。
  - 支持删除确认。
  - 支持从当前页后快速新增下一页。
  - 支持为 slide 设置名称，便于定位与编排。
- 图层管理增强
  - 支持图层重命名。
  - 支持图层锁定 / 解锁，避免误选误拖。
  - 支持图层隐藏 / 显示，便于复杂页面排查与排版。
  - 图层列表需能明确定位当前选中节点。
- 时间轴编排增强
  - 支持步骤重命名、排序、复制、删除。
  - 支持从当前步骤开始预览。
  - 支持一键重置当前 slide 播放状态。
  - 在现有触发模型基础上，补齐动画基础参数，至少包括 `duration` 与 `delay`。
- 本地图片导入
  - 支持直接选择本地图片文件导入，而不只依赖 URL。
  - 导入后的图片在项目持久化后仍可恢复。
  - 图片导入流程应尽量简单，减少额外配置成本。
- 页面背景配置
  - 支持按 slide 设置背景色。
  - 支持按 slide 设置背景图。
  - 背景图需支持基础填充方式，例如铺满或适配。
