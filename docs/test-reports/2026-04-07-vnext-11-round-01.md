## 测试结果

### 测试 1：工具栏图片入口命名统一

- 测试状态：通过
- 执行路径：
  1. 清空浏览器 `localStorage`，进入项目列表页
  2. 新建空白项目并进入编辑态工作台
  3. 读取编辑器工具栏插入区文案
- 测试结果：
  - 插入区文案为 `插入 文本 矩形 图片 图片框`
  - 存在单一“图片”入口与“图片框”入口
  - 未再出现旧的“导入图片”命名
  - 当前桌面视口下工作台未出现横向溢出

### 测试 2：工具栏本地图片插入

- 测试状态：通过
- 执行路径：
  1. 在空白项目中通过工具栏“图片”选择本地测试图片 `vnext08-sample.png`
  2. 等待属性面板切换到“组件属性”并完成自动保存
  3. 读取 `localStorage` 中当前项目首个 slide 的图片节点
- 测试结果：
  - 导入完成后画布立即出现图片节点，右侧面板展示“图片属性”
  - 自动保存完成后，图片节点字段正确写回：
    - `name = vnext08-sample`
    - `x = 480`
    - `y = 270`
    - `width = 320`
    - `height = 180`
    - `props.alt = vnext08-sample.png`
    - `props.objectFit = contain`

### 测试 3：属性面板直接更换图片

- 测试状态：通过
- 执行路径：
  1. 在“基础属性”中把当前图片节点改为：
     - `x = 420`
     - `y = 210`
     - `width = 360`
     - `height = 240`
  2. 等待自动保存后，在“图片属性”中点击“更换图片”
  3. 选择第二张本地测试图 `vnext09-bg-portrait.png`
  4. 再次等待自动保存并读取当前项目节点数据
- 测试结果：
  - 更换图片后，节点 `id` 未变化：
    - 变更前后均为 `node-966351ff-ba02-420d-8d36-21acabc2708d`
  - 节点位置与尺寸保持不变：
    - `x = 420`
    - `y = 210`
    - `width = 360`
    - `height = 240`
  - 图片资源已切换成功：
    - `props.alt` 从 `vnext08-sample.png` 变为 `vnext09-bg-portrait.png`
    - `props.src` 发生变化
  - `props.objectFit` 继续保持为 `contain`

### 测试 4：刷新后的持久化恢复

- 测试状态：通过
- 执行路径：
  1. 从编辑态切换到“预览”模式
  2. 等待预览画布渲染完成
  3. 读取预览 `lower-canvas` 上图片区域的顶部与底部像素
- 测试结果：

## 关键观察

- 当前 `VNext-11` 把“导入图片”收敛为统一的“图片”入口后，工具栏语义更直接，和“图片框”职责区分也更清晰。
- 换图链路当前是“替换资源而不重建节点”，这对保留排版结果非常关键；本轮通过节点 `id` 不变、位置尺寸不变和刷新后仍保持一致，验证了这一点。
- 更换图片后，节点名称仍保持第一次导入时的 `vnext08-sample`，而 `alt` 已更新为新文件名；本轮未将其判定为缺陷，因为需求中未要求自动重命名，但后续如果产品希望图层名称始终反映最新资源，建议单独确认规则。
