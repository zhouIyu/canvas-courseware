# Canvas Courseware VNext Plan

本文档用于记录下一版本的规划、范围边界和开发批次，不直接替代当前执行中的 `docs/TODO.md`。

## 版本定位

- 上一版本已于 2026-04-10 结束并归档，详见 `docs/RELEASES.md` 中的 `Release 05`。
- `VNext-31` 轻量诊断日志能力已完成开发与自测，后续若需微调日志字段或查看方式，以增量优化处理，不再作为下一批主线范围。
- 下一版本建议切换为 `VNext-32`，聚焦“架构治理与可持续演进”。
- 下一版本目标是在不扩张 MVP 功能范围的前提下，优先降低核心模块复杂度、控制本地资源持久化膨胀风险，并为后续 schema 升级和版本兼容预留标准化入口。
- 下一版本继续遵守 `docs/PRD.md` 中定义的架构边界，尤其保持 UI 层与事件层解耦、文档模型与引擎解耦。

## 核心治理范围

- [x] VNext-32-A 编辑器壳层与工作台职责拆分
  说明：
  重点拆分 `packages/vue` 与 `apps/playground` 中持续膨胀的壳层文件，降低单文件和单函数复杂度，让页面管理、画布布局、右键菜单、持久化与预览联动职责各自收敛。
  需求拆分：
  1. 将 `CoursewareEditor.vue`、`useCoursewareEditor.ts`、`ProjectWorkspacePage.vue` 中的高耦合逻辑拆为更清晰的子组件、composable 或应用层 service。
  2. 收敛编辑器舞台缩放、右键菜单、页面管理、自动保存与导入导出等独立职责，避免继续堆叠在单一入口组件中。
  3. 拆分后保持现有编辑 / 预览主链路、缩略图回写、自动保存和模式切换行为不回归。
  当前状态：
  已完成工作台顶栏、项目持久化、编辑器舞台、壳层动作与命令 API 的职责拆分；当前环境缺少 `node / pnpm`，本轮暂未实际执行 `typecheck` 与 `build`，待后续环境补齐后补跑静态基线。

- [ ] VNext-32-B Fabric 适配层职责拆分
  说明：
  聚焦 `packages/fabric` 中编辑态与播放态适配器的职责膨胀问题，按渲染同步、选中态恢复、文本编辑、播放调度与动画执行等边界拆开，降低后续引擎适配和问题定位成本。
  需求拆分：
  1. 拆分 `editor-adapter.ts` 中的画布同步、选中态保留、文本编辑和右键菜单接线逻辑。
  2. 拆分 `player-adapter.ts` 中的播放调度、步骤推进、触发器处理和动画执行逻辑。
  3. 保持适配层对 `@canvas-courseware/core` 标准命令与事件协议的单向依赖，不把 Fabric 细节泄漏到 Vue UI 层。

- [ ] VNext-32-C Schema 解析 / 校验 / 迁移层治理
  说明：
  当前文档导入导出能力已可用，但 schema 解析、运行时校验与版本兼容逻辑仍集中在单文件中。下一批需先把解析层拆清，再为未来版本演进预留稳定入口。
  需求拆分：
  1. 将 `document-io.ts` 按 parser、validator、migrator 等职责拆分，减少单文件复杂度。
  2. 在保持当前 `1.0.0` schema 正常解析的前提下，补齐标准化的版本迁移入口，避免后续版本升级只能“硬拒绝导入”。
  3. 保证节点、步骤、动画与触发对象之间的引用闭环校验继续稳定可用。

- [ ] VNext-32-D 图片资源存储模型升级
  说明：
  当前本地图片以 data URL 写入文档并最终落入 `localStorage`，短期可用，但继续扩展会带来容量、保存性能和导入导出体积风险。下一批需先重构资源存储模型。
  需求拆分：
  1. 评估并落地图片资源与课件 JSON 解耦的本地存储方案，避免大体积 data URL 持续膨胀文档体积。
  2. 保持图片导入、换图、设为背景、自动保存、刷新恢复与导入导出链路的使用体验不劣化。
  3. 为后续接入更大资源类型或远端资源管理预留统一资产标识与读取入口。
