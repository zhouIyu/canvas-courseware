# Canvas Courseware Requirements Pool

本文档用于记录产品侧持续收集中的需求池，仅做收集、整理和待评审记录，不等同于已排期开发任务。

## 待评审需求

- [ ] 统一图片插入与编辑语义：去掉“图片”和“导入图片”的重复表达，插入入口统一命名为“图片”；若当前“图片类型”仅代表重复入口，则从新增组件类型列表中移除；选中图片组件后，可在组件属性中直接更换图片资源，而不是删除后重新插入
- [ ] 编辑区支持右键菜单能力：在编辑画布区域提供右键快捷操作入口，减少频繁往返顶部工具栏、组件列表和属性面板的操作成本；具体菜单项与触发范围待补充确认
- [ ] 图片支持一键设为页面背景：在上传图片流程或图片右键菜单中，新增“设为背景”入口，让图片可直接成为当前 slide 背景；具体转换规则待补充确认
- [ ] 首页继续简化，移除或弱化首页中的大标题区域，降低首屏非必要信息占比
- [ ] 新增项目时，支持在创建流程中直接设置项目名称和页面尺寸，而不是创建后再补充配置
- [ ] 编辑页右侧删除“页面设置”模块，仅保留当前仍有价值的编辑与编排面板内容
- [ ] 文字元素支持双击直接进入编辑态，减少修改文本内容时的操作路径
- [ ] 将工具类高频功能外化到更方便操作的位置，而不是继续放在“组件列表”内部
- [ ] 组件列表改为编辑器右上角的独立浮层，支持点击展开 / 收起，默认收起为一个 icon 入口
- [ ] 保存或切换到下一页时，自动截图当前 slide 并更新左侧对应页面封面，保证缩略图与当前内容同步
- [ ] 组件列表支持拖拽排序，并直接通过排序结果修改组件层级
- [ ] 优化组件列表、组件属性与时间轴之间的联动交互，减少切换面板、重复配置和多余操作，形成更顺畅的编辑链路

## 需求细化记录

### RQ-IMG-01 统一图片插入与属性编辑链路

- 背景：
  当前界面同时出现“图片”和“导入图片”两类表达，用户难以理解两者差异；图片插入后，如果想替换素材，现有链路需要删除旧图片再重新插入，操作重复且容易丢失排版结果。
- 目标：
  统一图片相关入口命名，减少认知成本；补齐图片组件在属性面板中的换图能力，让用户在不破坏现有排版的前提下完成素材替换。
- 用户场景：
  用户在编辑课件时，需要插入一张本地图片；插入后发现素材选错，或想替换成另一张图，希望直接在组件属性里更换，而不是重新创建一个图片组件。
- 功能范围：
  插入区只保留一个“图片”入口，不再同时出现“图片”和“导入图片”两种重复表达。
  原“导入图片”入口如承担本地文件选择能力，则保留该能力，但对外统一命名为“图片”。
  若当前“图片类型”本质上只是重复入口，而不是独立组件能力，则从新增组件类型列表中移除重复项，但保留底层图片组件能力。
  选中图片组件后，右侧“组件属性”面板提供“更换图片”操作，支持重新选择本地图片资源。
  更换图片后，默认保留当前组件的位置、尺寸、旋转、层级、步骤归属和动画配置，不要求用户重新排版。
- 非功能约束：
  图片插入与更换操作的命名需在工具栏、右侧面板和相关提示文案中保持一致。
  更换图片后，项目本地持久化、再次打开项目和预览播放链路应继续可用。
  该需求仅调整图片插入与编辑链路，不扩展为远程 URL 图片管理、图片裁剪或图片素材库能力。
- 验收标准：
  用户在新增组件时，只会看到一个“图片”入口，不再看到重复的“导入图片”命名。
  用户插入本地图片后，可在“组件属性”面板中直接执行更换图片操作。
  更换图片后，原图片组件不需要删除重建，基础排版与编排配置保持不变。
  更换后的图片可正常保存、刷新恢复，并在预览模式下正常显示。
- 风险与依赖：
  待实现阶段确认现有“图片”与“导入图片”在代码中的真实职责边界，避免误删仍承担有效能力的入口。
  待实现阶段确认图片替换后是否需要补充尺寸适配策略，例如保持原尺寸、按新图比例自适应或提供手动重置入口。

### RQ-CTX-01 编辑区右键菜单

- 背景：
  当前编辑区的高频操作主要分散在顶部工具栏、组件列表和属性面板中。如果用户希望在当前鼠标位置附近直接完成操作，现有链路可能偏长。
- 目标：
  为编辑区补充右键菜单入口，让用户在画布内完成更顺手的快捷操作，降低操作跳转成本。
- 用户场景：
  用户在编辑画布中选中对象或停留在某个位置时，希望通过右键快速打开相关操作，而不是每次移动到固定工具区寻找入口。
- 功能范围：
  在编辑区提供右键菜单能力。
  右键菜单的具体功能项、排序、分组和是否区分“空白区域 / 选中对象 / 多选对象”等状态，当前待确认。
  当前仅记录为编辑态需求，不默认扩展到预览态。
- 非功能约束：
  右键菜单的交互语言应与现有工作台的命名和操作保持一致。
  若接管浏览器默认右键菜单，需要在可用性和误触风险之间进一步评估。
- 验收标准：
  编辑区可触发产品定义的右键菜单入口。
  菜单中展示的操作项与最终确认的需求范围一致。
- 待确认项：
  菜单是否只在选中对象时出现，还是空白区域也有独立菜单。
  首批菜单项包含哪些操作，例如复制、粘贴、删除、层级调整、对齐、锁定、隐藏，以及图片对象上的“设为背景”等。
  是否需要根据单选、多选、图片、文字等不同对象类型展示不同菜单项。
  是否完全屏蔽浏览器默认右键菜单，或提供条件性保留策略。

### RQ-BG-IMG-01 图片一键设为页面背景

- 背景：
  当前图片上传和页面背景配置是分开的操作链路。如果用户希望把一张上传的图片直接作为当前页面背景，现有流程可能需要先插入图片，再进入页面背景配置，操作较绕。
- 目标：
  为图片补充“设为背景”快捷入口，降低把图片应用为页面背景的操作成本。
- 用户场景：
  用户上传一张图片后，希望直接将其作为当前 slide 背景使用。
  用户已经在画布中插入了一张图片，希望通过右键快速把这张图片设置为当前 slide 背景。
- 功能范围：
  在上传图片相关流程中提供“设为背景”入口，允许用户将所选图片直接应用为当前 slide 背景。
  在图片对象的右键菜单中提供“设为背景”入口。
  该能力默认作用于当前 slide，不默认扩展到整个文档。
  图片设为背景后的对象处理方式当前待确认，例如保留原图片对象、删除原图片对象，或由用户二次确认。
- 非功能约束：
  “设为背景”的命名需与页面背景配置中的术语保持一致。
  该需求依赖页面背景能力支持图片背景，且不默认扩展为图片裁剪、滤镜或素材库管理能力。
- 验收标准：
  用户在上传图片流程中可看到“设为背景”入口，或在图片对象右键菜单中可触发“设为背景”操作。
  用户执行“设为背景”后，当前 slide 背景可切换为目标图片。
  设置结果可被正常保存、刷新恢复，并在编辑态和预览态正常显示。
- 待确认项：
  上传图片时，“设为背景”是独立按钮、二次选择，还是上传后的快捷操作。
  对已有图片执行“设为背景”后，原图片对象是否保留在画布中。
  设为背景时是否需要同时提供填充方式选择，例如铺满、包含、拉伸。
  该入口是否同时出现在上传流程和图片右键菜单中，还是二选一即可。
