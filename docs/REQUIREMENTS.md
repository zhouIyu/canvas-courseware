# Canvas Courseware Requirements Pool

本文档用于记录产品侧持续收集中的需求池，仅做收集、整理和待评审记录，不等同于已排期开发任务。

## 待评审需求

- [ ] 新增“轻量诊断日志能力”：为编辑器关键链路提供统一结构化日志，优先服务开发排障与一致性复盘，暂不扩展为远程采集、运营分析或用户审计系统

## 需求细化记录

### RQ-LOG-01 轻量诊断日志能力

- 背景：
  当前编辑器在图片导入、图片替换、设为背景、自动保存、时间轴配置等关键链路中，错误反馈和诊断信息较为分散，存在仅修改状态、不保留错误细节，或只通过零散 `console.error` 输出的问题。
  当出现“保存失败但难以复现”“页面设置与预览回显不一致”“时间轴步骤异常”“图片资源处理失败”等问题时，开发和测试缺少统一、可回溯的排障依据。
- 目标：
  为课件编辑器补充一套轻量、统一、可结构化读取的诊断日志能力，用于辅助开发排障、测试复盘和关键链路一致性定位。
- 用户场景：
  开发人员在排查自动保存失败、图片处理失败、页面背景设置异常、时间轴状态不一致等问题时，希望能查看最近一次关键操作的统一日志，而不是只能依赖零散控制台输出或手工复现。
  测试人员在提交缺陷或复测问题时，希望能结合关键操作日志更快确认问题发生在哪一条链路。
- 功能范围：
  新增统一日志接口，支持 `debug / info / warn / error` 等基础级别。
  首批记录范围聚焦关键链路，包括：命令执行、自动保存、导入导出、图片资源处理、页面背景写入、时间轴关键变更。
  日志应支持携带最小上下文信息，例如 `projectId`、`slideId`、`nodeId`、事件名称、执行结果和错误摘要。
  默认优先服务开发态诊断，至少支持控制台输出和最近若干条关键日志的内存保留。
  当前默认不扩展为远程日志上报、运营埋点、用户行为分析或审计日志系统。
- 非功能约束：
  日志能力必须遵守 UI 层与事件层解耦原则，不能让 UI 组件直接演变为分散的日志拼装入口。
  日志默认不记录整份课件 JSON、大体积图片数据或高噪音重复内容，避免影响性能和可读性。
  日志方案应便于后续扩展，但当前版本应保持轻量，不引入重型平台依赖。
- 验收标准：
  图片导入 / 替换 / 设为背景失败时，能够记录统一格式的错误日志，并包含最小上下文信息。
  自动保存成功或失败时，能够记录统一格式的关键日志，并区分执行结果。
  页面背景写入与时间轴关键变更时，能够记录可用于复盘的关键操作日志。
  开发人员能够在开发态查看最近一次关键链路日志，而不需要临时在多个文件中手工加 `console`。
  当前日志方案不包含远程采集、用户行为审计和运营分析能力。
- 风险与依赖：
  如果日志记录范围过大，可能带来性能负担和排障噪音，因此需要优先收敛首批记录边界。
  如果日志上下文定义不统一，后续会形成新的维护成本，因此需要先明确统一字段模型。
  该能力依赖现有命令层、保存链路和关键编辑事件具备可插入的统一记录点。
- 当前规划结论：
  该需求暂不纳入当前已确认的下一版本 `VNext-25` 至 `VNext-30`。
  后续将以单独技术任务的方式进入后续版本规划，而不是在现有产品功能任务中顺带落地。
- 待确认项：
  “最近若干条关键日志”的保留方式是否需要补充为可导出 JSON 的诊断能力。
